# k() 公式最佳实践：10 大常见坑

> 基于 JSA880 框架 100+ 次 bug 修复积累的实战经验。读完这份文档，你的 k() 公式报错率降低 50%。

---

## 坑 1：中文逗号 — `f1，f2` 会被整段当一列

### 问题

WPS 中文输入法环境下，用户很容易用中文逗号（`，`）替代英文逗号（`,`）。例如：

```
=k("JSA.superPivot", A1:H40, "f3，f2", "f6", "count()")
```

中文逗号 `f3，f2` 不会被拆分为两列，而是被当作一个字段名 `"f3，f2"` 整体处理，导致解析失败。

### 正确写法

始终使用英文逗号 `,`：

```
=k("JSA.superPivot", A1:H40, "f3,f2", "f6", "count()")
```

### 受影响范围

| 参数 | 示例 | 后果 |
|------|------|------|
| `rowFields` / `colFields` / `dataFields` | `"f1，f2"` | 整段当一列，字段解析失败 |
| `colSelector` 列选择器 | `"f1，f2，f3"` | 不拆分，只匹配一列 |
| `resultSelector` 输出选择器 | `"sum(f3),count()"` → `"sum(f3)，count()"` | 聚合项不拆分 |
| 自定义标题 | `"商品，国家"` | 标题整体填入一个 cell，不拆分到多 cell |

> **注意：** 自定义标题（如 `['f1,f2', '商品，国家']`）中的中文逗号从 v4.0.37 起已被框架兼容。但参数入口（公式字符串内）的逗号必须用英文逗号。

## 坑 2：fN 格式 vs 数字索引 — 两种都支持但混用会乱

### 问题

k() 公式的参数传递有两种风格：

- **fN 格式**（公式友好）：`"f3,f2"` 表示第 3 列和第 2 列
- **数字索引**（JS 原生）：`[2, 1]` 表示索引 2 和 1（从 0 开始）

两种在框架内部最终都转换为相同的 `parseFieldsConfig` 处理。但如果在一个公式里混用两种格式，会造成字段顺序混乱。

### 示例

```js
// ❌ 混用：rowFields 用 fN，colFields 用数字索引
=k("JSA.superPivot", A1:H40, "f3,f2", [5], "count()")  // 不推荐

// ✅ 统一使用 fN 格式
=k("JSA.superPivot", A1:H40, "f3,f2", "f6", "count()")

// ✅ 统一使用数字索引
=k("JSA.superPivot", A1:H40, [2,1], [5], "count()")
```

### 规则

- **WPS 公式里传字符串用 fN**（`"f3"`）— 这是框架设计的主要路径
- **VBA/JSA 代码里传数组用数字索引**（`[2]`）
- **不要在同一个函数调用里混用两种格式**

## 坑 3：header 处理 — `headerRows=0` 意味着没有表头

### 问题

k() 公式创建 `Array2D` 时，默认 `headerRows=0`，意味着**所有行都是数据行**。如果你传递的数据第一行是表头：

```
数据:
[["产品", "价格", "数量"],
 ["A", 10, 100],
 ["B", 20, 200]]
```

那么 `=k("JSA.superPivot", A1:C3, "f1", "", "count()")` 会把 `"产品"` 也当成分组值，输出会包含一行 `"产品" -> 1`。

### 正确做法

有以下几种方式标明表头：

**方式 1：使用 `headerRows` 参数**

```js
=k("JSA.superPivot", A1:C3, "f1", "", "count()", 1)
// 第 6 个参数 headerRows=1 表示跳过第 1 行
```

**方式 2：使用 Array2D 构造时设 `_header` 属性**

```js
var arr = Array2D([["产品","价格","数量"], ["A",10,100], ["B",20,200]]);
arr._header = ["产品", "价格", "数量"];
```

> 框架内部会自动检测 `_header` 属性并跳过表头行（v3.7.9+）。
> 链式调用时 `_header` 会通过 `_new()` 方法自动传递。

## 坑 4：lambda vs 字符串 — 两种 callback 形式

### 问题

k() 公式支持两种 callback 写法：

| 形式 | 示例 | 性能 | 兼容性 |
|------|------|------|--------|
| **Lambda 表达式** | `x => x.f2 === 'A'` | ⭐ 更好 | ⚠️ 要求 WPS 2021+ |
| **字符串表达式** | `"x.f2 === 'A'"` | ⭐ 一般 | ✅ 所有版本 |

Lambda 使用 `eval()` 执行，性能更好但依赖 ES6 箭头函数支持。字符串表达式需要经过 `parseLambda` 解析，有额外开销。

### 常见错误

```js
// ❌ Lambda 里引用了外部变量（WPS JSA 环境 eval 作用域受限）
var threshold = 100;
=k("(x=>x.f2 > " + threshold + ")", A1:C10)  // 可能报错

// ✅ 字符串形式嵌入变量
=k("x.f2 > 100", A1:C10)

// ✅ 使用 fN 列选择器
=k("f2>100", A1:C10)  // f2 自动展开为 _[1]>100
```

### 规则

- **简单场景用字符串**：`"f2>100"`、`"x.f1+x.f2"` — 稳定可靠
- **复杂逻辑用 Lambda**：`x => x.f2 > 100 && x.f3 < 50` — 但要确保 WPS 版本支持
- **$0, $1, $2 索引语法**：`"$0>$1"` — 自动转为 `_[0]>_[1]`，兼容性好
- **fN 列选择器**：`"f1+f2"` — 自动转为 `_[0]+_[1]`

## 坑 5：NaN/null 处理 — 聚合函数的跳过规则

### 问题

聚合函数对 `NaN` 和 `null` 的处理不是完全透明的，不同函数行为不一致：

| 函数 | 行为 |
|------|------|
| `z求和` / `sum` | 跳过 NaN/null，浮点结果四舍五入到 1e-10 |
| `z平均值` / `average` | 分母**只计数有效数值项**，不包含 NaN/null |
| `z最大值` / `max` | 纯数字列跳过 NaN；字符串列按 `String <` 比较；混合列按字符串比较 |
| `z最小值` / `min` | 同上 |
| `count()` | 计数所有非空行 |

### 陷阱示例

```js
// 数据: [10, null, 20, NaN, 30]

// sum = 60 (null 和 NaN 被跳过)
=k("JSA.sum", A1:A5)

// average = 60/3 = 20 (只计 3 个有效值,不是 5)
=k("JSA.average", A1:A5)

// max 字符串列 ["苹果","香蕉","橘子"] → "橘子" (按 String 比较,不是 undefined)
=k("JSA.max", B1:B3)
```

### 规则

- **空数组**调用聚合函数返回 `undefined` 或 `0`（视具体函数而定）
- **字符串列**聚合 `max`/`min` 使用字符串比较（v4.0.37+ 支持）
- **不要依赖**聚合函数自动处理所有边缘情况，有疑问用 `z筛选` 预处理

## 坑 6：JSON.stringify 边界 — `undefined` 的特殊行为

### 问题

JavaScript 原生 `JSON.stringify` 对 `undefined` 的处理：

```js
JSON.stringify(undefined)      // undefined (不是字符串 "undefined"!)
JSON.stringify([1, undefined, 2])  // '[1,null,2]' (数组内 undefined 变 null)
JSON.stringify({a: undefined})     // '{}' (对象属性被删除)
```

框架内部多处使用 `JSON.stringify` + `JSON.parse` 做数据转换（例如 WPS host array → JS array），如果数据中包含 `undefined` 值，可能导致意外的 `null` 替换或属性丢失。

### 实际影响

- WPS 单元格空值传给 JSA 时可能是 `undefined`，经 `JSON.stringify` 往返后变成 `null`
- `null` 在 WPS spill 时显示为 `0`（而不是空白），影响视觉效果
- 链式调用中 `Array2D` 实例经过 `JSON.stringify` 后失去类型信息

### 规则

- **不要依赖 `undefined` 的语义** — 在 JSA880 框架中 `undefined` 不可靠
- **空值用空字符串 `""`** — 框架内部 v4.0.33 起已把 `null` 补丁为 `""`
- **传递 WPS Range 时**，框架会自动 `_toRealArray()` 转换，不必手动 `JSON.stringify`

## 坑 7：z去重 性能 — O(unique) + O(N) proxy 开销

### 问题

`z去重`（`distinct`）内部使用 `Set` / callback + proxy 机制：

```js
Array2D(arr).z去重('f1')           // 按第1列去重，返回完整行
Array2D(arr).z去重('f1,f2')         // 按第1+2列组合去重
Array2D(arr).z去重('f1', 'f2,f3')   // 按f1去重，输出f2,f3列
```

### 性能特性

- **时间复杂度**：O(N) — 使用 `Set` 或 WeakSet 去重（线性扫描）
- **proxy 开销**：每行创建一个 fN proxy（`x.f1`, `x.f2`），约 5-10μs/行
- **大数据集（>10K 行）**：proxy 创建开销显著，建议用 `z分组汇总` 代替去重+聚合

### 对比

```js
// ❌ 慢：先 z去重 找出唯一产品，再链式聚合
=k("$$.distinct('f1').filter(...)", A1:E50000)

// ✅ 快：直接用 z分组汇总 内置聚合
=k("JSA.groupInto", A1:E50000, "f1", "sum(f3),count()")
```

### 规则

- **<1K 行**：`z去重` 随意用，性能无感
- **1K-10K 行**：可以用，但注意 proxy 开销
- **>10K 行**：优先用 `z分组汇总`（`groupInto`），避免链式去重

## 坑 8：z超级透视 多列 col 字段 — 中文/数字混类型乱序

### 问题

`z超级透视`（`superPivot`）对列字段（`colFields`）的值做排序时，如果列值同时包含数字和字符串（例如 `[2021, 2022, "总计", "X"]`），排序结果可能不符合预期。

### 根因

v4.0.42 之前的比较器对混合类型使用 `localeCompare`，导致：

```
["X", 2021, 2022, "Z"] → ["X", "Z", 2021, 2022]  // 字符串在前，数字在后
```

### 正确行为（v4.0.42+）

- 纯字符串 → `localeCompare`（字典序）
- 纯数字 → 数值比较
- 混合类型 → **保持输入顺序**（不再用 `localeCompare` 越界）

### 规则

- **依赖 colFields 排序时，确认所有值类型一致**（要么全数字，要么全字符串）
- **如果必须混合，用手动排序预处理数据**
- **使用 `sort` 符号明确控制**：`colFields="f6+"` 为升序，`colFields="f6-"` 为降序

## 坑 9：fN 范围 (f3-f5) — 用户手写时易写中文短横

### 问题

框架支持 `f3-f5` 范围语法，自动展开为 `f3,f4,f5`。但中文输入法下用户可能输入中文破折号 `—`（U+2014）或半角破折号 `–`（U+2013）：

```
=f3—f5    // ❌ 中文破折号，不会被识别为范围
=f3-f5    // ✅ ASCII 连字符，正确展开
```

### 正确写法

始终使用 ASCII 连字符 `-`（键盘上 `0` 右边的键）：

```js
=k("JSA.z多列排序", A1:H40, "f3-f5+")     // 按 f3-f5 范围升序
=k("$$.distinct('f3-f5')", A1:H40)          // 取 f3-f5 范围去重
```

### 受影响函数

| 函数 | fN 范围支持 |
|------|-------------|
| `z多列排序` | ✅ 支持 `f3-f5+` |
| `z去重` resultSelector | ✅ 支持 `f3-f5` |
| `z选择列` colSelector | ✅ 支持 `f3-f5` |
| `z超级透视` rowFields/colFields | ❌ 不支持范围，必须逐个写 |

## 坑 10：z映射 header 行 — 默认会把 header 也算进去

### 问题

`z映射`（`map`）和 `z筛选`（`filter`）默认处理**所有行**，包括表头行。如果你的 `Array2D` 有 `_header` 属性：

```js
var arr = Array2D([["产品","价格"], ["A",10], ["B",20]]);
arr._header = ["产品", "价格"];

// z映射 会对表头行也执行 mapper，导致 "产品" * 2 = NaN
arr.z映射(function(row) { return row[1] * 2; });
```

### 正确做法

**方式 1：设置 `_header` 属性（v3.7.9+ 自动跳过）**

```js
arr._header = ["产品", "价格"];
arr.z映射(function(row) { return row[1] * 2; });  // 自动跳过第1行
```

> 框架内部检测到 `_header` 属性存在时，`z映射`、`z筛选`、`z排序` 自动 `skipHeader=1`。

**方式 2：手动传入 `skipHeader`**

```js
arr.z筛选(function(row) { return row[1] > 10; }, 1);  // skipHeader=1
```

**方式 3：链式调用中保持 `_header`**

```js
var result = Array2D(data)
    .z筛选(function(row) { return row[1] > 10; })  // 自动跳表头
    .z映射(function(row) { return row[0]; });       // 自动跳表头
```

### 规则

- **创建 `Array2D` 时设 `_header`**，后续链式方法自动识别
- **`z超级透视` 输出自带 `_header`**，可以直接链式 `z筛选`/`z映射`
- **手动构造的数组**没有 `_header`，需要显式传 `skipHeader`

## 快速排错清单

在 WPS 单元格输入 `=k("JSA.k.help")` 可以看到内置排错指南。常见错误速查：

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| `#K_ERR: pos=0, FN` | fn 字符串解析失败 | 检查函数名拼写、引号配对 |
| `#K_ERR: pos=1, INTERNAL` | 框架未加载 | 加载 JSA880.js 加载项 |
| `#K_ERR: pos=0, TYPE` | 类型错误 | 传了不该传的对象类型 |
| `#NAME?` | ThisWorkbook 缺少 wrapper | 粘贴 3-5 行 wrapper 代码 |
| 输出只有一行 | Array2D 实例未 unwrap | 用 `.val()` 取纯数组 |

### 自检公式

在任意单元格输入以下公式，返回 `1,2,3,4,5` 表示框架正常：

```
=k("JSA.getIndexs", 1, 5, 1)
```
